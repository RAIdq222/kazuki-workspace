# サーバー化・複数人利用 設計メモ（ドラフト）

> ステータス: 構想〜設計。相談ベースの合意を反映。実装前のたたき台。
> 前提となる単一ユーザ版の設計は `docs/design-notes.md` を参照。

## 0. このメモの位置づけ

現状の「背景原図 修正支援ツール」（単一マシン・CLIバッチ・ファイルベース）を、
**複数人の美術監督が使えるサーバー基盤**へ拡張するための設計。
将来的に美観チェック・カット制作・作画・制作進行管理などの工程へ広げることを
前提に、**後付けの工程拡張が"二度作り"にならない骨組み**を最初に定義するのが目的。

## 1. 決定事項（相談で合意した割り切り）

- **同時編集はしない**。同じカット/資料を2人が同時に触ることは運用で避ける。
  → 必要なのは「編集中ロック」＋「状態のライブ反映」で足りる。CRDT等の共同編集は不要。
- **本丸は"カスケード再生成"**。上流資料（美術ボード等）の差し替えが多数のカットに波及する。
  「いつ・どの条件で再生成を走らせるか」が設計の核。
- **制作進行管理は自作前提**。「何が管理された状態か／報告・相談がどう回るか」は
  現場でチューニングが要りドメイン依存。既存ツールは"はまるモジュールだけ"連携で使う。

## 2. 背骨 — 3つの土台（全機能がこの上に乗る）

再生成カスケードも進捗管理も工程拡張も、下記3点の組み合わせで表現できる。
ここをMVPの骨として最初に入れる。

```
① 依存・来歴グラフ  … 何が・何の何版から作られ、何に影響するか（provenance + DAG）
② 工程の状態機械    … 未着手 → 作業中 → 確認待ち → OK / リテイク（＋担当者）
③ ルールエンジン    … ①②の状態を監視し、stale化・通知・エスカレーションを起こす
```

- 「ボード差し替えで再生成対象を洗い出す」= ① → ③
- 「報告・相談の自動化」= ② → ③
- 「承認済みは勝手に作り直さない」= ① × ②

## 3. カスケード再生成の設計原則

一言原則：**「stale判定は自動 / 再生成の実行は人間ゲート / 結果は新版・非破壊」**。
自動でやるのは"気づき"まで。実費と判断が絡む実行は人間。結果は積む（上書きしない）。

### 3.1 来歴(provenance)の記録
すべての生成結果に「入力の版」を紐づけて残す。これが無いと影響範囲を機械的に出せない。

```
生成結果#123 = 原図v3 + 美術ボードv7 + プロンプトv2 + モデル設定v1
```

上流の版が上がったら、その版を入力に持つ生成結果をグラフで辿って抽出できる。

### 3.2 stale判定（自動）
上流Assetの新版が確定したら、それを入力に持つ下流の生成結果／カットを
**「要再確認(stale)」に自動マークするだけ**。自動で再生成はしない（コスト＋どのみち人が確認する）。

### 3.3 再生成の実行ポリシー（人間ゲート）
軸は3つ。既定値を下記に置く（現場で調整）。

| 軸 | 選択肢 | MVP既定 |
|---|---|---|
| トリガ源 | ボード/原図/香盤表 差し替え・プロンプト改訂・モデル設定変更・リテイク指示 | 全て stale 化の対象 |
| タイミング | 即時手動 / バッチ（夜間まとめ） / 段階ゲート | 選択式手動 ＋ 夜間バッチ投入 |
| 影響範囲 | 全dependentsをstale化して人が間引く / 変更分類で自動で絞る | 全stale化 → 人が間引く（保守的） |

**段階ゲート**（重要）: カットの状態で再生成の扱いを変える。
- ドラフト段階のカット → 再生成は気軽にOK
- 承認済みのカット → 再生成には承認者の許可を要する（勝手に作り直さない）

### 3.4 非破壊・版管理
- 再生成は**新リビジョンを作るだけ**。承認済みの版は残す。
- 「採用(current)」ポインタは**人が更新するまで旧版のまま**。
- 旧版 vs 新版の比較UIを用意（採否判断のため）。

### 3.5 担当者への波及（同期との接続）
ボード差し替えはAさんの操作でも、stale化するカットの担当はB・C・Dさん。
→ ③ルールエンジンが「あなたの担当カットN件が要再確認になった」を該当者へ通知。
依存グラフ → タスク → 通知、で繋ぐ。

## 4. 「管理されている状態」の定義（4分解）

「何がまとまり、適切に報告・相談されるか」を4つに分けて実装する。

1. **可視化（正本）**: カット状態 → 話数進捗 → 作品進捗が一望でき、担当・詰まり・遅延が見える。
2. **例外検知＋エスカレーション（＝報告・相談の核心）**:
   全部を報告せず、**人間の判断が要るものだけを適切な人へ自動で上げる**。ルール例：
   - リテイク3回超え → 人手送りフラグ（単一ユーザ版から継承）
   - AIの充足判定が「自信なし」
   - ボード差し替えで stale がN件超え
   - 承認待ちが所定日数滞留 / 締切に対し遅延
   → 「pull（ダッシュボード）」と「push（通知）」の両方で出す。
   → **このルール群が自社独自の核**。抽象から決めず、現場の実ルールから逆算して育てる。
3. **判断の記録**: なぜこのボードを使ったか / なぜ人手送りか / 誰がOKを出したか。
   来歴(3.1) ＋ レビュー・承認履歴で担保。
4. **計画・キャパシティ**: 誰が何件をいつまでに。既存ツールが強い領域 → 連携 or 軽量実装。

## 5. 初期データモデル（案）

工程を汎用の第一級概念として持つのが要点。原図修正も美観チェックも作画も、
`Task` の"種別違い"として同じ骨に乗せる。

### エンティティ
```
Organization
 └─ User               … 所属ユーザ
 └─ Project(作品)
     ├─ Membership     … User × Project × Role（作品単位の権限スコープ）
     ├─ Episode(話数)
     │   └─ Cut(カット) … cut_no, 状態(current_state), 現採用リビジョン(current_revision_id)
     │       └─ Revision … カットの版。生成結果 or 手作業成果を指す
     ├─ Asset          … 実体ファイル参照（原図/美術ボード/香盤表/生成結果/PSD）
     │   └─ AssetVersion … version, storage_key, checksum, uploaded_by
     ├─ Task(工程タスク) … type(genzu_fix|beauty_check|cut_prod|sakuga|…),
     │                      state, assignee, cut_id?, input_asset_ids[], output_asset_id?
     ├─ Dependency     … from_asset_version → to_generation（来歴DAGの辺）
     ├─ GenerationJob  … model_id, params, input_versions[], status, cost, requested_by
     ├─ ReviewEvent    … approve/retake/comment, actor, target_revision, timestamp
     └─ Notification / EscalationEvent … ルールエンジンの出力
AuditLog                … 生成実行・承認・権限変更などの監査（誰がいつ何を）
```

### 主要な状態
- **Cut.state / Task.state**: `未着手 → 作業中 → 確認待ち → OK`（＋ `リテイク`, `人手送り`）
- **Revision.role**: `draft` / `candidate` / `approved` / `superseded`
- **Cut ↔ stale**: current_revision の入力版より新しい上流版があれば stale フラグ

### 権限（RBAC）
- ロール例: 管理者 / 美術監督 / 作画監督 / 制作進行 / 閲覧者
- スコープ: Organization 全体 / Project 単位 / Episode 単位
- アクション権限: 閲覧 / 編集 / **承認** / **生成実行（＝実費発生）**
- **コストガバナンス**（複数人×複数作品で必須）:
  - User/Project ごとの生成クォータ・予算上限
  - 生成ジョブの監査ログ（誰がいつ何カット回したか＝AuditLog）
  - APIキーはサーバー側で秘匿（クライアントに置かない）

## 6. 技術スタック（1人MVP：BaaS土台 ＋ 自作の核）

1人でMVPを成立させる鍵は、**認証・ストレージ・リアルタイム・行レベル権限を自前で書かない**こと。

### 既製で済ませる（BaaS / 外部モジュール）
- **BaaS（Supabase 想定）**: Postgres＋Auth＋Storage＋Realtime＋RLS（行レベル権限）。
  → 5節の DB・認証・アップロード・ライブ反映・作品単位権限の"配管"の大半をカバー。
- **オブジェクトストレージ**: BaaS Storage or S3/GCS/R2。presigned URLでクライアント直PUT
  （既存の Higgsfield `media_upload`+PUT と同じパターン）。
- **通知配信**: Slack（接続済み）／メール。エスカレーションの出力先は自作しない。
- **ファイル保管・閲覧の橋渡し**: Google Drive / Box（接続済み）。
  ただし**正本メタはDB**、実体だけ預ける。
- **締切**: Google Calendar 連携。
- **初期の報告ビュー（裏技）**: 自前ダッシュボードを作る前に、状態DBを Notion データベースへ
  同期するだけで「一望できる表」が手に入る。可視化(4-1)の初期実装を省ける。

### 自作する核（ここが独自価値・外注も既製品も効かない）
- **依存・来歴グラフ**（provenance ＋ DAG、stale判定）
- **工程の状態機械**（汎用 Task）
- **ルールエンジン**（例外検知・エスカレーション）
- **生成ワーカー**（既存 `image_aspect.py`・香盤表パーサ・Higgsfield呼び出しを内包）

### egress について
実サーバーではサンドボックスの egress 制限が無いため、
単一ユーザ版で詰まっていた `upload.higgsfield.ai` へのPUTは素直に通る（むしろ楽になる）。

## 7. フェーズ計画（縦切りで育てる）

1. **薄い縦切り**: 原図修正だけを、数人の美術監督が使えるWebサービスに。
   BaaS土台 ＋ ただし**背骨①②③は最初から仕込む**（後で二度作りしないため）。
2. **実データで実需を確定**: 同期粒度・権限・エスカレーションルールを現場で調整。
3. **工程を足す**: 美観チェック → カット制作 → 作画 … を `Task.type` として追加。
4. **制作進行**: 最後 or 既存ツール連携。独自価値（AI×美術工程）に集中。

「全部入りを設計してから作る」より、動く縦切りを早く出して育てる。
