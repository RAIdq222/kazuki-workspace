# 背景原図 修正支援ツール — 設計メモ（ドラフト）

> ステータス: 要件整理〜PoC段階。レビュー必須。精度検証は別セッションで実施予定。

## 1. 目的

商業アニメの背景原図（美術への依頼用ラフ）を、美術ボードを参照しつつ、
指示内容（建物・自然物の配置、キャラ位置）を制約として守ったまま、
パースの狂いや設定との齟齬を画像生成で修正する。

- 生成物は**最終納品ではなく、美術監督が確認・修正する叩き台**（ただし品質は納品レベルを志向）。
- 1話あたり約300カットを一括処理したい。
- 入力PNGは **PSD原図からの自動書き出しに対応**（§10、`src/genzu_fix/psd_export.py`）。
  レイヤー構造は不定形だが、保存済みの「表示状態」をそのまま合成すればよいので自動化できる。

## 2. 使用モデル / 確認済みの事実

- 画像生成: **Higgsfield 経由の GPT Image 2**（model_id: `gpt_image_2`）。
- 確認済みスキーマ:
  - 画像入力（編集）対応。tags に `editing` あり。
  - 解像度ティア: `1k` / `2k` / `4k`（デフォルト 1k）。
  - 品質: `low` / `medium` / `high`。
  - 比率（離散・整数比のみ）: `1:1, 4:3, 3:4, 16:9, 9:16, 3:2, 2:3`。
- 未確定（要・お試し検証）:
  - 入力画像を**2枚同時**（原図＋美術ボード）渡せるか。不可なら美術ボードは言葉で渡す。
  - 「配置を保ったままパースを直せるか」の実効性能。

## 3. MVP スコープ

「**指示プロンプト作成 ＋ 生成実行**」まで。検証ループ・PSD格納は後段。

```
原図PNG  ─┐
          ├─► ① 原図理解(GPTで構図/配置/指示書きを把握。指示文字・補助線は無視)
香盤表    ─┤
美術ボード ┘
              ▼
          ② プロンプト統合（場面設定 + 守る制約 + 直す指示 + 画風参照）
              ▼
          ③ 比率・解像度の事前調整（余白パディングで許容比率へ）  ← 実装済み
              ▼
          ④ GPT Image 2 で生成（Higgsfield）
              ▼
          ⑤ 余白を切り戻して元画角に復元 → 保存
```

## 4. 香盤表（Excel）について

列構成: `カットNO / 時間 / 場所 / 3Dモデル / 天気 / 光源 / 色味 / キャラ / 服装 / 小物 / 2D / 3D / 備考`。
セクション: `アバンpart / OP / Apart / CM / Bpart`。

- 場面設定（時間・天気・光源・色味・場所）はこの表から機械的に取得できる。
- カットNO は `3～14`（レンジ）, `2、3`（リスト）, `欠番`(備考), `回想`注記が混在
  → カット番号→行の解決ロジックが必要（レンジ展開・欠番除外）。
- 場所名（例 `南康郡_街`, `住宅街(隔離空間)`）と美術ボードのファイル名は厳密に紐づかない
  → 「場所名 → ボードファイル」対応表を作品ごとに1回作る（半手動）。
- 原図の欄外にも `BG:場所名` が書かれており、香盤表との突合の手がかりになる。

## 5. 原図理解で見るべき観点（v1 仮説 / 育てる対象）

実物（南康郡の街並み／一点透視）から判明した重要点:
**原図には「絵そのもの」以外に、指示・補助の描き込みが乗っている。**
（手書き指示 `淡色処理`、欄外管理情報、パース補助線・VPマーク・フレーム枠・タップ穴）
→ 「直す対象の絵」と「指示・補助線」を**分けて扱う**工程が必須。

観点（初期案）:
- 構図種別（一点/二点透視 等）、消失点・地平線の位置、画角
- 主要構成物の**配置**（左右・奥行き・前後関係）= 守るべき制約
- キャラの立ち位置（描かれる場合）= 守るべき制約
- パースの収束ズレ、様式的な正しさ（建築様式など）= 直す候補
- 美術ボード（画風・色味・時代設定）との齟齬 = 直す候補
- 欄外/画面内の指示書き・補助線の識別（無視対象）

## 6. 比率・解像度の事前調整（実装済み）

`src/genzu_fix/image_aspect.py`

- 原図の縮尺・比率は変えない。**余白(パディング)**で許容比率へ寄せる。
- 許容比率の中から**余白が最小**になる比率を自動選択。
- 復元は**相対座標(割合)**で行い、出力比が要求比とズレても正しく切り戻す。
- 例: 原図 2000×1391 → `3:2` を選択、左右に43px余白 → 生成 → 余白除去で元画角へ。

## 7. 運用前提

- 1カット約5分（プロンプト3分＋生成2分）想定、300カットで約25時間 → 夜間バッチ。
- カット単位で中間成果物をファイルに残し、**途中再開可能**にする。
- リテイクは最大3回（充足判定はAI、後段で実装）。失敗カットは人手送りフラグ。

## 8. 実行経路 / ネットワークの知見（重要・検証済み）

Higgsfield への接続経路を切り分けた結果:

- **MCPコネクター経由の操作は許可リスト不要で動く**（Anthropicサーバー経由）。
  実際に通った: `balance`(残高), `models_explore`(仕様), `media_upload`(presigned URL発行),
  `generate_image`(命令発行), Google Drive からの原図ダウンロード。
- **サンドボックスから直接出る通信は環境の egress 許可リストで制限される**。
  → 以前は `upload.higgsfield.ai` への PUT が `403` でブロックされていた。
  → **【2026-06-19 解消・実証】** 環境の Network access に Higgsfield を許可後、
     `upload.higgsfield.ai` への直 PUT が **200** で成功。`media_upload`→PUT→`media_confirm`→
     `generate_image`→`show_generations`(結果URL取得)→ダウンロード→切り戻し までの
     **全工程を web セッション内で完結**できることを確認済み。
- `media_import_url`（サーバー側fetch）は **Google Drive 各URL形式でリダイレクト超過(>2)で失敗**。
  Drive は import 非対応とみなす。直リンク配信のホストが必要。
  → ただし `media_upload`+PUT が web 上で通るようになったため、import_url は不要になった。

### 結論（実行方針）
- **本番の300カットバッチ**: ユーザー環境（Mac/サーバー）で Python＋**Higgsfield CLI** 実行。
  egress 制限が無いため `media_upload`+PUT もCLIも素直に通る。これが本筋。
- **Claude Code on the web で動かす場合**: 環境の Network access に Higgsfield を許可
  （`*.higgsfield.ai`, `upload.higgsfield.ai`, `*.cloudfront.net` ＋既定リスト、または Full）。
  → **2026-06-19 に許可済み・動作実証。web 上でお試し〜小バッチが完結可能。**

## 10. PSD→PNG 書き出し（実装済み・2026-06-19）

`src/genzu_fix/psd_export.py`（`psd-tools` 使用）。手動の「表示レイヤー書き出し」を自動化。

- `export_visible_to_png(psd, out, bg=(255,255,255))` … **本命**。PSD に保存された
  合成プレビュー＝**表示レイヤーの WYSIWYG** をそのまま PNG 化（高速・忠実）。`bg=None` で透過維持。
- `list_layers(psd)` … レイヤー一覧（名前/可視/種別/bbox/階層）を取得。
  実 PSD で「絵」と「指示・補助/チェック」の分離構造が見えた（例: `用紙` `100Fr赤`(枠)
  `tap`(タップ) / `LO` 配下の `【演出】【監督】【作監】【総作監】` チェック群 / `線画` `ラフ`）。
  → 原図理解 v1 の素性データとして有用。
- `export_with_overrides(psd, out, show=, hide=)` … レイヤー名で可視を上書きして再合成
  （`.visible` 書き換え＋`composite(force=True)`。ネスト層にも効く）。
- `insert_result_layer(psd, image, out_psd, base_name="AI原図修正")` … **最終成果物**。
  生成結果(PNG)を元PSDの**最前面に新規レイヤーとして差し込み**、別PSDへ保存（入力は非破壊）。
  元レイヤー構成は全保持。**リテイクは枝番**で積む（`AI原図修正` → `AI原図修正_02` → `_03`…）。
  検証済み: 保存→再オープンでレイヤー存在・元レイヤー保持・合成は最新を最前面に反映。

### 生成結果のPSD差し込み（insert_result_layer）の要点
- psd-tools 1.17 は書き込み可能: `PixelLayer.frompil(img, parent, name=, top=, left=)`＋`PSDImage.save()`。
- **日本語レイヤー名**: PSD レガシー名は macroman で書くため日本語は不可。
  `frompil(name=...)` はレガシー名へ直書きして save 時に `UnicodeEncodeError`。
  → 回避: frompil 後に `layer.name = "AI原図修正"` と**セッター経由**で代入すると、
  レガシー名は `"?"`、Unicode 名は `luni` タグブロックへ格納され、**Photoshop は luni を表示**。
- frompil は parent 末尾＝最前面に追加。差し込み層はフルキャンバス不透明なので最前面が見える。
- 入力PSDは変更せず必ず `out_psd` に保存（非破壊）。本番では出力を別名/別ディレクトリに。

### 運用で詰める予定（パターン収集後に判断）
- `export_visible_to_png` で**引継ぎメモ等のテキストレイヤー/参考資料が一緒に写る**ケースがありうる。
  まず実運用でパターンを集め、除外ルール（`kind=='type'` の除外や名前規則）を後で決める。
  `list_layers` が `kind` を返すのでテキスト層の洗い出しには使える。

### 検証で分かった psd-tools の注意点
- `composite()` は既定で**保存済みプレビューを返す**（可視変更が反映されない）。
  上書きを効かせるには `composite(force=True)` で再レンダリングが必須。
- `composite(layer_filter=...)` は**トップレベルのレイヤーにしか呼ばれない**（ネスト不可）。
  そのため名前指定の on/off は layer_filter ではなく `.visible` 書き換え＋`force=True` で行う。
- 「枠・タイトル・タップ」等が **背景ラスター(`LO用紙`)に焼き込まれている**ことがあり、
  名前レイヤーを隠しても完全には消えない。完全な「絵だけ抽出」は後段の原図理解で扱う。

## 14. 重要な前提の修正（2026-06-22・レビュー指摘より）

サンプル生成のレビューで判明した、設計の前提に関わる指摘。**未解決のまま本番に進まないこと。**

### 14.1 「主体に属する/環境に属する」の線引き（QCの no_subject 定義）
- 誤: 「固定物は残す／可動物は消す」。可動か固定かは無関係。
- 正: **主体（キャラ）に属するものを消す**（持つ/身につける/連れている物・人物由来の影/効果）。
  **環境に属するものは残す**（棚の本は残す／人物が持つ本は消す）。
- 判断には**絵コンテ・美術ボードの文脈が要る**ことが多い。作品ごとに美術側と定義する。

### 14.2 美術ボード・絵コンテをプロンプトに使えていない（現状の穴）
- ep7サンプルの生成は**ボード未参照・絵コンテ未参照の“当て推量”**だった。
- 本来 §3 の通り、①原図理解 ②プロンプト統合 で **絵コンテ（その場面が何か）＋美術ボード（時代/様式/構成）**
  を読み込むべき。ここが未接続。ep7はボード未確定・絵コンテが92MB(未取得)で、まだ素材も揃っていない。

### 14.3 キャラ遮蔽カットの「背景補完」は捏造になる（手法の限界）
- キャラが画面下を覆う構図（膝上など）では、**隠れた地面/背景は原図に描かれていない**。
- そこから人物を消して生成補完すると、**AIが背景を捏造**する（根拠なし）。例: 233-④の地面・下草。
- 正しい運用:
  - **BGonly素材を使う**（スタジオは `*_BGonly` を作っている＝背景は別に用意される前提）。
  - BGonlyが無ければ、隠れた背景は**絵コンテ＋美術ボードに基づき意図的に設計**する領域であり、
    インペイントで埋める対象ではない。
- → **本ツールの「人物除去インペイント」はキャラ遮蔽カットには不適**。
  適用対象は「背景が主役のカット」「BGonly素材があるカット」に絞る。カット選別の判定基準に反映する。

## 15. 探索で分かったこと（2026-06-22・続き）— コンテ/注記の活用と方針の再修正

### 15.1 大容量ファイルは「リンク共有」でサンドボックス直取得・直読みできる（実証）
- 原図フォルダを共有済みのため、`gdrive.py` で **絵コンテ92MB を直接DL**でき、`PyMuPDF` で
  任意ページをレンダリングして**コンテの中身を読める**（カット番号は手書き＝画像、テキスト層は無い）。
- → 「コンテ/ボードから情報を得る」は web セッションでも**実際に可能**。大判PSDも同様に取得可。

### 15.2 §14.3 の「キャラ遮蔽カットは対象外」は行き過ぎ。撤回する
- 正しい姿勢: カットを選別して逃げない。人物は消すが**アイレベル・空間・プレーン構造を保てば**解ける。
- 重要: **原図の注記がプレーン(BOOK)構造と撮影意図を教えてくれる**。
  例 cut233(C.233): 立ち人物＋森。画面下は地面ではなく**「手前のモヤ BOOK1」＝別の前景霧プレーン**。
  右隅に**回転/ぼかしの撮影メモ＋円形図**（コンテ233の「DF:ぼかし/速度線」と一致）。
  → BG原図として正しいのは「**森プレーンを描き、下部は霧BOOK用に開けておく**／人物・付随物は消す」。
  　以前の④が描いた“草地”は誤り（そこは地面でなく霧プレーン領域）。
- 学び: 単純インペイントでなく、**注記＝プレーン分け・撮影指示・アイレベルを読んで反映**すれば
  キャラ遮蔽カットも背景の叩き台にできる。情報（コンテ/ボード/注記）が増えるほど精度が上がる。

## 16. 品質バーの指摘（2026-06-22・ユーザーレビュー）— 線質と「品位の引き上げ」

サンプル出力への重要な批評。**ここを満たさないと“使える原図”にならない。**

### 16.1 線質（最優先課題）
- 問題: 出力が**一定太さのクリーンな輪郭＝塗り絵/ベクター調**で、**入り抜き（線の入り・抜き・強弱）**や
  手描きの揺らぎが無い。「AI絵」感が強い。
- 要件: 熟練美術の鉛筆クリーンアップのように、**近景/輪郭は太め・遠景/内部は細く繊細・植生や質感は
  軽い擦れ/破線**で、**線幅変調と入り抜き**を持たせる。均一輪郭・CG的な無機質さを避ける。
- 注意: GPT Image 2 は放置すると均一ベクター線に寄る。**線質は言葉で明示強制**しないと出ない
  （プロンプトに「hand-drawn / line-weight variation / 入り抜き / not vector / not coloring-book」）。
  それでも限界がある場合は、線画特化モデルや後処理(線幅変調)も検討。

### 16.2 「転写」でなく「品位の引き上げ」
- 問題: 元原図の低品質をそのまま残し、人物/記号を消す“整理作業”に留まっている。
- 要件: パースの狂い・弱い構造を**直して絵として成立させる**。様式的に正しい構造ディテールを
  控えめに足す。＝「背景だけの絵を作る」のではなく**作画品質を上げる**のが目的。

### 16.3 プロンプトはシーン固有に厚く書く（ユーザー実例が有効）
- ユーザーの良好な結果は、**具体的なシーン記述**で出ていた: 構図(例「2-1-2構成・5棟」)を名指しで
  維持、舞台設定(中国南北朝・山岳集落・国境地帯)、各建物を格上げしない指示、使ってよい/避ける
  植生・生活道具の列挙、雨設定の表現方法、出力=整理線画。
- → §13 の「作品共通＋シーン固有＋カット固有」テンプレに、**構造の名指し維持**と**線質指定**を
  必須項目として組み込む。アイレベルだけでなく §15.2 の項目群＋線質をワンセットで。
