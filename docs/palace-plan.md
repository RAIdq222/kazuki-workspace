# 大宮殿(紫禁城風)3Dステージ化 計画書

対象: 中軸線上に門・正殿・脇殿・回廊が連なる大宮殿ボード。複数建物+内装あり。
敷地想定 約250×400m(従来シーンの100倍超の面積、路地=3×42m比で線形30倍・面積1000倍級)。

## 0. 基本方針: 面積100倍 ≠ トークン100倍

コストは**面積ではなく「ユニークな細部の数 × 検証往復の回数」に比例**する。
宮殿建築は反復の塊(柱間・斗栱・欄干・瓦・回廊)なので、

1. **キット・オブ・パーツ**(パラメトリック部品ライブラリ)を先に作り、
2. 全建物をキットの組み合わせで記述し、
3. インスタンス化(リンク複製)でデータ量を抑える

ことで、従来シーン1本の**8〜15倍程度**に収められる見込み。
竹林(リンク複製346本→HTML 3.1MB)で実証済みのアプローチの拡張。

## 1. なぜ「全体像→各建物ディテール」の2段階では足りないか

その順序は正しいが、間に**キット化**の工程を挟まないと破綻する:

- 屋根の反り・斗栱・欄干を建物ごとに手書きすると、20棟×数百行=破綻
- 様式の修正(「屋根の反りが強すぎる」等)が20棟に波及して修正コストが爆発
- 椅子の歪み問題と同型: 早期に1棟で様式を確定させないと、後からの直しが最も高くつく

正しい順序: **配置図 → キット → ヒーロー1棟で様式確定 → 全域マッシング → 各棟ディテール → 内装**。

## 2. 段階的構築プラン

### Phase 0: マスターレイアウト(配置図)
- `src/stage3d/palace/layout.py`: 中軸線グリッド上に全建物の footprint / 向き / 階数 / LOD等級をデータ定義
- 座標系: 中軸線=Y軸(奥が+Y)、原点=正面門中央。建物は名前付きアンカー(gate_south, hall_main, …)
- 箱マッシング(箱+単純屋根)の俯瞰レンダーで**構図をユーザー承認** ← ゲート①
- ボードの画角再現カメラもこの時点で決める(構図が合わないと全部やり直しのため)

### Phase 1: パーツキット
- `src/stage3d/palace/kit/`: roofs.py / structure.py / ornament.py / textures.py
- 部品: 反り屋根(寄棟/入母屋/重檐)、基壇(須弥座)、柱列、斗栱(簡略LOD)、欄干、階段、格子窓・扉、回廊ユニット、香炉等の点景
- テクスチャ: PIL手続き生成(瓦葺き・朱壁・木部・石畳・格子)を共有ライブラリ化
- **正殿1棟をヒーロービルド**して様式(反り具合・色・ディテール密度)を**ユーザー承認** ← ゲート②
  - 味の修正はここで吸収するのが最安。20棟に展開後の様式変更は10倍高い

### Phase 2: 全域マッシング(LOD1)
- 全棟をキットの組み合わせで配置。軸線対称・基壇の高さ関係・中庭の抜けを検証
- 遠景の借景建物(画面奥の街並み)は box+屋根 の LOD0(路地シーンの far houses と同じ手法)
- 俯瞰+ボード画角+逆方向の3点で**空間の正しさを検証**(現実的であること優先の原則)

### Phase 3: 各棟ディテール
- `src/stage3d/palace/buildings/`: 1棟=1モジュール(hall_main.py, gate_south.py, corridor_east.py, …)
- `build.py --only hall_main` の**選択ビルド**で、1棟だけ再生成→検証の高速反復
- ディテール棟は5〜8棟に絞る(ボードで大きく写る棟)。残りはキット標準のまま
- 棟単位で独立しているので**並列サブエージェント(worktree)分担が可能** — 壁時間短縮

### Phase 4: 内装
- ヒーロー2〜3室(正殿の玉座の間など)。1室=従来の屋内シーン1本分の規模
- 内装は建物モジュール内の `interior()` 関数として実装、`--lod 2` で有効化
- ビューワーには扉位置の**テレポートプリセット**(外→内ジャンプ)を追加

### Phase 5: ビューワー統合
- インスタンス化で geometry を共有(欄干・柱・斗栱)。フォグで遠景を溶かす
- エリアジャンプ(門前/中庭/正殿前/玉座の間…)のプリセット群
- サイズが1ファイルの限界を超える場合: 全域LOD1ビューワー + ヒーロー棟/内装の個別ビューワーに分割

### Phase 6: 検証・仕上げ
- `lint_scene.py`: footprint重なり / 軸対称ずれ / 階段と扉の整合 / 基壇高さの一貫性 を自動チェック
  (規模的に目視だけでは漏れるため、空間の正しさをスクリプトで担保)
- コンタクトシート一括レンダー(全棟×3アングル、低サンプル)+ Playwrightプリセットスイープ
- 最終Cyclesレンダーは決めアングルのみフル品質(CPU 4コアの壁時間対策)

## 3. トークン見積(実測ベース)

実績: 従来シーン1本 = スクリプト213〜548行 + テクスチャ生成〜250行、
修正コミット2〜13回/シーン、実消費 約30〜50万トークン(検証レンダー往復込み)。

| Phase | 内容 | 見積(万トークン) |
|---|---|---|
| 0 | マスターレイアウト+承認用マッシング | 15〜30 |
| 1 | パーツキット+ヒーロー正殿(様式確定) | 60〜100 |
| 2 | 全域マッシング LOD1 | 20〜40 |
| 3 | 各棟ディテール 5〜8棟 × 5〜15万 | 50〜100 |
| 4 | 内装 2〜3室 × 20〜40万 | 60〜120 |
| 5 | ビューワー統合・性能調整 | 20〜40 |
| 6 | 検証・lint・最終レンダー | 30〜60 |
| **計** | | **約250〜500** |

- 最大の不確実要素は Phase 1 の反り屋根(試行錯誤が読みにくい)と Phase 4 の内装点数
- ユーザー修正往復(椅子問題のような様式フィードバック)は1往復あたり+5〜20万
- 並列エージェント分担は壁時間を縮めるがトークンは+2〜3割(調整オーバーヘッド)

## 4. 承認ゲート(手戻り最小化のため)

| ゲート | タイミング | 確認内容 |
|---|---|---|
| ① 配置 | Phase 0 末 | 俯瞰マッシング+ボード画角ラフ。「構図・建物の数と配置はこれでいいか」 |
| ② 様式 | Phase 1 末 | 正殿1棟のフルディテール。「屋根の反り・色・密度はこれでいいか」 |
| ③ 全域 | Phase 2 末 | 全域ビューワーラフ。「歩き回った感じの空間スケールはこれでいいか」 |
| ④ 内装 | Phase 4 各室 | 室ごとに従来どおりの確認 |

## 5. ディレクトリ構成(予定)

```
src/stage3d/palace/
  layout.py        # マスター配置データ (全棟のfootprint/向き/LOD)
  build.py         # CLI: --only <棟> --lod 0|1|2 --time ...
  kit/
    roofs.py       # 反り屋根 (寄棟/入母屋/重檐)
    structure.py   # 基壇/柱列/壁/回廊ユニット
    ornament.py    # 斗栱(簡略)/欄干/階段/格子/点景
    textures.py    # PIL: 瓦/朱壁/木部/石畳/格子
  buildings/
    hall_main.py   # 正殿 (exterior + interior)
    gate_south.py  # 正面門
    ...
```
