# 大宮殿(紫禁城風)3Dステージ化 計画書

対象: 中軸線上に門・正殿・脇殿・回廊が連なる大宮殿ボード。複数建物+内装あり。
敷地想定 約250×400m(従来シーンの100倍超の面積、路地=3×42m比で面積1000倍級)。

検証済み: 本計画は3つの独立調査(宮殿建築の寸法標準 / Three.jsビューワーのスケール限界実測 /
計画の敵対的レビュー)を通して修正済み。SwiftShaderの描画コストと既存GLBのバイト/triは実測値。

## 0. 基本方針: 面積100倍 ≠ トークン100倍

コストは**面積ではなく「ユニークな細部の数 × 検証往復の回数」に比例**する。
宮殿建築は反復の塊(柱間・斗栱・欄干・瓦・回廊)なので、

1. **キット・オブ・パーツ**(パラメトリック部品ライブラリ)を先に作り、
2. 全建物をキットの組み合わせで記述し、
3. リンク複製+glTFインスタンス化でデータ量を抑える

ことで従来シーン1本の**8〜15倍程度**に収める。竹林(リンク複製346本→HTML 3.1MB)の拡張。

**ただし律速はトークンではなく壁時間**(Cycles CPU 4コアのレンダー往復)。
反復確認はビューワーのPlaywrightスクショと低解像度クロップレンダーに限定し、
Cyclesフルレンダーは承認ゲート直前のみ、という運用規約を守ることが見積の前提。

## 1. 順序の考え方: 「全体像→ディテール」に3つの工程を差し込む

素朴な2段階では破綻する。理由と対策:

- **屋根曲面は質的に新しい技術課題**(既存stagelibはbox/cyl/plane等のみ。反り屋根はbmesh直組みが必要)
  → 様式承認より前に「屋根1枚だけのスパイク」で技術方式を確定する
- **ビューワー/エクスポートの限界は最後に発覚すると手遅れ**(現行build_viewer.pyはexport_apply=Trueで
  メッシュ共有が消える。無対策だと単一HTMLが数十MB級に)
  → 全域マッシングの時点でGLB→HTML→FBXまで貫通させ、サイズ予算をそこで確定する(トレーサービュレット)
- **様式の修正は展開後だと10倍高い**(椅子問題と同型)
  → ヒーロー正殿1棟で様式を確定してから20棟に展開する

正しい順序:
**配置図+lint → 屋根スパイク → キット+ヒーロー正殿(様式確定) → 全域マッシング貫通(予算確定) → 各棟ディテール → 内装(別シーン) → 統合仕上げ**

## 2. 段階的構築プラン

### Phase 0: マスターレイアウト+資料確定+lint基盤
- `palace/layout.py`: 中軸線グリッド上に全建物の footprint / 向き / 階数 / LOD等級をデータ定義。
  座標系: 中軸線=Y軸(奥+Y)、原点=正面門中央、1BU=1m、地面z=0
- **資料確定**: ボードに写る棟/写らない棟を区分し、写らない棟の方針(実紫禁城の縮退寸法 or
  GPT Image 2で補助ボード生成)をユーザーと合意。寸法基準表(§5)もここで承認
- **`lint_scene.py` をこの時点で実装**(最終検証ではなく常時実行): footprint重なり / 左右対称ずれ /
  階段と扉の整合 / 基壇高さの一貫性 / 回廊端点の座標一致(接続チェック)
- 箱マッシングの俯瞰レンダー+ボード画角ラフで**構図をユーザー承認** ← ゲート①
- カメラ: `add_camera` の clip_end を敷地対応(1000m)にパラメータ化

### Phase 0.5: 屋根スパイク(技術検証、上限を切って実施)
- 反り屋根(歇山=入母屋)1枚だけを作る使い捨て検証。bmeshグリッドロフト方式(§6)を実証
- `--style line`(線画)レンダーでボードの屋根シルエットと直接重ねて比較
- ここで方式が固まらない限りPhase 1に進まない(計画全体がこの1点に依存するため)

### Phase 1: パーツキット+ヒーロー正殿
- `palace/kit/`: roofs / structure / ornament / textures(§5の15関数)
- **生成方式の規約**: テンプレートメッシュ+`object.copy()`リンク複製(竹林方式)または
  bmesh一括生成を必須。bpy.ops直接生成は棟あたり少数に制限(1棟のビルド時間を秒単位に保つ)
- マテリアルは**PIL画像テクスチャ+uv_scaleタイリングを標準**にする(プロシージャルノードは
  GLB/FBXでフラット色に落ちるため、ビューワーが主役の本件ではCycles専用の上乗せに限定)
- 斗栱・欄干は3段LODを最初から仕様に組み込む(§6の2位)
- **正殿1棟をヒーロービルド**し、マテリアル様式シート(瓦色・朱色・彩画テクスチャ見本)と
  合わせて**様式をユーザー承認** ← ゲート②

### Phase 2: 全域マッシング+パイプライン貫通(トレーサービュレット)
- 全棟をキットで配置(LOD1)。軸線対称・基壇の高さ関係・中庭の抜けを検証(lint常時実行)
- **必ずフルパイプラインを通す**: GLB(EXT_mesh_gpu_instancing)→gltfpack(meshopt)→単一HTML→
  Playwrightスクショ→FBX往復。ここでポリゴン予算(§4)とHTML/FBXサイズを実測し、
  単一 vs エリア分割の最終判断を下す
- ビューワー改修もここで先行実装: camera.far のCFG化(1500)、meshoptデコーダ組込、
  線画モードのInstancedMesh除外
- 全域俯瞰レンダー+**中間ビューワーHTMLをユーザーの実機で確認** ← ゲート③
  (SwiftShaderはソフトウェアGLなので性能の最終判断はユーザー実機でしかできない)

### Phase 3: 各棟ディテール
- `palace/buildings/`: 1棟=1モジュール、`build.py --only <棟>` の選択ビルドで高速反復
- **キットAPIをバージョン付きで凍結**してから着手(並列分担の前提)。オブジェクト/マテリアル名は
  棟プレフィックス必須、`mat()` に同名異パラメータ検出を追加
- ディテール棟は5〜8棟(ボードで大きく写る棟)。統合ビルド+lint合格を各棟の受け入れ条件に
- 並列サブエージェント(worktree)分担可。ただしユーザーレビューは直列なので短縮効果は限定的

### Phase 4: 内装(独立シーンとして)
- ヒーロー2〜3室(玉座の間など)。**全域シーンに同居させず、従来どおり1室=1シーン**
  (別blend/別HTML)で作る — 既存パイプラインの強みをそのまま使う
- 全域ビューワー側は簡易内装(床+壁色)+扉位置のエリアジャンプで接続
- 室ごとに従来どおりユーザー確認 ← ゲート④

### Phase 5: ビューワー統合・仕上げ
- Phase 2で骨格は済んでいる前提で、エリアジャンププリセット群・時間帯ライティング・
  フォグ調整・テクスチャアトラス最適化
- ビューワーconfigのカメラプリセットは layout.py から座標変換込みで自動生成(手書き転記は
  20棟分になると事故るため禁止)

### Phase 6: 総ざらい検証
- 既存lintの総ざらい+Playwrightプリセットスイープ(全域はLOD1状態か棟単体HTMLで)
- コンタクトシートは**棟単位の分離シーン**で低サンプルレンダー(全域フルレンダーの乱発は壁時間で死ぬ)
- 最終Cyclesレンダーは決めアングルのみ: 全域俯瞰1枚+中軸線3枚+内装3枚 程度

## 3. トークン見積(検証・修正済み)

実績: 従来シーン1本 = スクリプト213〜548行+テクスチャ〜250行、修正コミット2〜13回/シーン、
実消費 約30〜50万トークン。敵対的レビュー後の再配分:

| Phase | 内容 | 見積(万トークン) | 壁時間の目安 |
|---|---|---|---|
| 0 | レイアウト+資料確定+lint基盤 | 20〜35 | 半日 |
| 0.5 | 屋根スパイク | 5〜15 | 半日 |
| 1 | キット+ヒーロー正殿(様式確定) | 80〜130 | 2〜3日(レンダー往復含む) |
| 2 | 全域マッシング+パイプライン貫通 | 30〜50 | 1〜2日 |
| 3 | 各棟ディテール 5〜8棟 | 50〜100 | 2〜4日(並列化で短縮可) |
| 4 | 内装 2〜3室 × 20〜40万 | 60〜120 | 2〜3日 |
| 5 | ビューワー統合・仕上げ | 40〜80 | 1日 |
| 6 | 総ざらい検証・最終レンダー | 15〜30 | 1日(レンダー待ち主体) |
| **計** | | **約300〜550** | **実働2週間前後** |

- 従来シーンの8〜15倍。ユーザー様式フィードバック1往復あたり+5〜20万を別枠で見る
- 並列エージェント分担は壁時間を縮めるがトークンは+2〜3割
- 中間ゲートを飛ばして手戻りが出た場合は550万を超え得る(ゲート①〜③が保険)

## 4. ポリゴン/サイズ予算(実測ベース)

| 項目 | 予算 | 根拠 |
|---|---|---|
| 描画三角形(全域) | 650k以下(理想500k) | SwiftShader実測 0.7ms/1k tri: 500k=0.36s/フレーム、検証スクショ成立 |
| ユニーク三角形 | 120k以下 | 実測34B/tri → ジオメトリ4〜5MB、meshopt圧縮で1MB前後 |
| テクスチャ | 2.5〜3MB(512²アトラス基調) | 竹林実績1.44MBの約2倍 |
| 全域ビューワーHTML | **8MB以下** | GLB 3.5〜5.5MB×1.33+JS0.6MB。無対策だと11〜15MB以上になる |
| 内装ビューワーHTML | 各2〜3.5MB | 既存実績どおり |
| 全域FBX | 15〜30MB | モディファイアを繰り返しパーツに残すと数倍化するので注意 |

**ファイル構成の結論: 「全域1本(簡略・内装なし・≦8MB)+内装は室ごとに別ビューワー+
必要なら正殿エリア詳細版1本」の4〜5ファイル構成。** 1ファイルに全部入れる案は非推奨。

技術要点:
- `export_gpu_instances=True`(EXT_mesh_gpu_instancing)をbuild_viewer.pyに追加 —
  Three.js GLTFLoaderが自動でInstancedMesh化、ノードJSON肥大も解消
- 圧縮は Draco ではなく **gltfpack(meshopt)**: デコーダ約30KBがesbuildにそのまま
  バンドルでき単一HTML・オフラインと相性が良い
- 欄干望柱150tri/本、柱は8角形、遠景棟3k tri/棟、瓦はテクスチャ+バンプでジオメトリ化しない
- Playwright検証は固定waitでなくレンダー完了フラグのポーリングに変更(ロード15〜30秒見込み)

## 6. 三大技術難所と攻略法

1. **反り屋根の曲面生成**(最難関): bmeshで u×v グリッドを数式生成してロフト。
   v方向(軒→棟)は挙架列を積分した凹カーブを8〜12分割、u方向は軒の矩形→棟の線分を線形補間
   (廡殿=寄棟はコーナー列を45°の隅棟線に吸着)。起翘は端部パラメータtに対し
   z += lift·t³(三次なので中央は水平を保つ)。歇山(入母屋)は同じグリッド生成器の分岐+
   両妻に山花板planeで実装できる。脊はポリライン→カーブ+ベベル。瓦はテクスチャで
2. **斗栱・高欄の物量制御**: 実物準拠だと斗栱数千攒・望柱千本超で即死。
   3段LOD(遠=PIL帯テクスチャ1枚 / 中=箱6個の簡略アセットのリンク複製 / 近=ヒーロー正面のみ
   詳細メッシュ)を最初から仕様に組み込み、カメラごとにLOD割当をlayoutのspecで宣言
3. **シーン規模管理**: キット→棟→シーンの3層ビルド。棟は個別アセット化しシーン側は
   インスタンスで組む。反復確認はクロップの部分レンダー+ビューワースクショで数分に保つ。
   遠景棟は張りぼて(正面テクスチャ板+簡易屋根)

## 6.5 レンダー自己レビュー規約 (納品前に毎回、初見の目で)

ユーザー指摘で発覚した「見た瞬間わかる」系の失敗を繰り返さないためのチェックリスト。
レンダーを送る前に全項目を目視で通し、引っかかったら直してから出す:

1. **接地**: 浮いている要素はないか (重檐の上層、欄干、部材同士の隙間の影)
2. **向き**: 階段は高い方に向かって上っているか / 建具・簾・看板の天地左右
3. **貫通・埋まり**: 人型や点景が地面・壇に埋まっていないか、部材の食い違い
4. **未テクスチャ面**: 側面・背面の貼り忘れ (色が浮いて見える面がないか)
5. **スケール**: 人型(1.65m)と比べて破綻していないか (段の高さ・手すり・扉)
6. **対称性**: 中軸対称のはずの要素がズレていないか
7. **カメラ**: 構図が壁・柱・点景に塞がれていないか。ボード角は原画と見比べる
実施ログ: 主殿ヒーローでは 欄干スラブ化/望柱z反転/上層側面未貼り/人型埋まり を
自己検出したが、**階段の逆向きは見逃してユーザー指摘で発覚** — 項目2はこの反省による。

## 7. 承認ゲート(手戻り最小化)

| ゲート | タイミング | 確認内容 |
|---|---|---|
| ① 配置 | Phase 0 末 | 俯瞰マッシング+ボード画角ラフ。「構図・建物の数と配置はこれか」 |
| ② 様式 | Phase 1 末 | 正殿フルディテール+マテリアル様式シート。「反り・色・密度はこれか」 |
| ③ 全域 | Phase 2 末 | 全域ビューワーを**ユーザー実機で**確認。「スケール感と動作は良いか」 |
