## 2. 「天使ちゃんスタジオ」の仕組みの棚卸し

天使ちゃんスタジオ = Higgsfieldアプリ（`summer-bell-707.higgsfield.app`、β版 2026-07-09リリース）。
「深夜のドカ食い天使ちゃん」チームの、企画〜生成〜編集用データ書き出しの一元管理ダッシュボード。
リポジトリを読んで確認した中核メカニズムは以下の7つ。**この案件でもそのまま流用できるもの◎、
改造して使うもの○、置き換えるもの×** を付す。

| # | 仕組み | 実体 | 応用 |
|---|---|---|---|
| M1 | **カード=制作単位の状態機械** | D1 `cards`。status: 企画案→承認(制作GO)→生成中→レビュー→完成/破棄。「承認=生成のGO、完成=検収OK」という意味論 | ○ カード=話数(エピソード)に置き換え。監修ステートを追加 |
| M2 | **ダッシュボード×AIワーカーの分業** | 人はカード作成・承認・レビューだけ。Claude運用セッションがカードを拾い、解析→台本(scenario)→カット分割→プロンプト設計→検証生成を行い、`worker_stage`（工程ラベル＋プログレスバー）で進捗を可視化。feedbackを書いてstatusを戻すと再投入 | ◎ そのまま。本案件のAI展開は「脚本→絵コンテ→プロンプト」 |
| M3 | **カット分割とプロンプト2層** | `cards.cuts` = [{n, label, duration, prompt}]。カット別プロンプトは概要版（チェック用・常時表示）と詳細版（実生成用・折りたたみ）の2層 | ◎ そのまま。絵コンテのカット表がここに載る |
| M4 | **参照画像の体系** | キャラ定型プロンプト（英文の固定記述）＋参照画像チップ（三面図・小物、デフォルトON）。生成はSeedance 2.0等のfnfジョブ、テイクごとに「採用」フラグ | ○ キャラが複数になる。キャラ・舞台の「承認済み設定」だけを参照に使う仕組みに拡張 |
| M5 | **シーンライブラリ（意味検索）** | 過去動画16本358シーンをタグ（場所/動作/表情/ショット/時間帯）＋FTS＋埋め込みで登録。「ジト目」「玄関で靴を脱ぐ」等の自然文で過去カットを検索→生成時の舞台参照にして場面を一致させる | ○ 過去動画が無いので「設定ライブラリ」として始め、承認済みカットを順次登録して育てる |
| M6 | **編集データ書き出し** | Premiere互換XML(xmeml)/EDL生成、`deliverables`（完成ファイル一覧） | ◎ そのまま |
| M7 | **企画の入口自動化** | トレンド取得→カード化、1行フラッシュアイデア→AIがフルショート案に展開 | × 置き換え。入口は「脚本取り込み（PDF→構造化→話数カード自動作成）」 |

技術スタック（流用可能）: TanStack Start + Cloudflare Workers + D1（additiveマイグレーション）、
Higgsfield SDK「fnf」（gpt-image-2 / nano-banana / seedance-2.0 / kling-3 / veo 等のジョブ投入・
メディア・コスト・確認ゲート）、quanta UI。認証はSign in with Higgsfield（各自のクレジットで生成）。

## 3. 全体アーキテクチャ（3層）

```
[愛知医大スタジオ]  Higgsfieldアプリ（天使ちゃんスタジオをフォーク）
  └ 状態の一元管理: 話数・カット・設定アセット・テイク・監修・書き出し（D1=唯一の製品状態）
[AIワーカー]        Claude運用セッション（複数並行可）
  └ 脚本解析 / 絵コンテ展開 / プロンプト設計 / 検証生成 / worker_stage報告
[git（このリポジトリ）]
  └ セッション間で共有すべき事実: 脚本テキスト・シリーズバイブル・設計・決定事項・スキル
```

天使ちゃん運用と同じく、**人（アイズ制作チーム・石橋さんら）はダッシュボードだけ触る**。
AIワーカーの思考・手順はgitに文書化し、どのセッションでも再現できるようにする（CLAUDE.mdの鉄則）。

推奨: **新規アプリとしてフォーク**（天使ちゃんスタジオはキャラ定型プロンプト・参照画像・9:16
YouTube前提がハードコードに近く、共用アプリ化は改修が大きい）。fnf/quanta/認証/書き出しは
そのまま持ってくる。

## 4. データモデル（D1・天使ちゃんスキーマからの差分）

```sql
-- 話数（cardsの置き換え。1カード=1話）
episodes(id, ep_no, title, script_md,          -- 脚本全文（整形済み）
         scenario,                              -- AIワーカーが書く演出台本（あらすじ+VO+演出意図）
         cuts JSON,                             -- [{n, label, duration, dialogue, camera,
                                                --   board_url,            -- 絵コンテ画（画像生成）
                                                --   prompt_summary, prompt_full,  -- 2層プロンプト(M3)
                                                --   ref_asset_ids[], status}]
         status,                                -- 下記ステータス機械
         feedback, worker_stage, deliverables JSON, final_video_url, ...)

-- 設定ライブラリ（scenesの置き換え+拡張。キャラデザ・美術・プロップの正）
assets(id, kind,          -- 'character'|'location'|'prop'|'style'
       name,              -- 例: 主人公 / スタチン長老 / プラーク火山(荒廃) / 王城・謁見の間
       description,       -- 日本語の設定文
       prompt_en,         -- 生成用の定型英文（天使ちゃんの「定型キャラ記述」方式・JP/EN対訳）
       media JSON,        -- [{label:'三面図'|'表情集'|'美術ボード'|..., media_id, url}]
       tags,              -- 検索用（場所/時間帯/天候 等。M5のタグ設計を踏襲）
       status,            -- 案 → 社内OK → 監修済（監修済のみ本番生成の参照に使える）
       version, created_at, ...)

-- 生成テイク（card_jobsと同じ。cut単位）
takes(id, episode_id, cut_n, job_id, kind,     -- 'board'|'still'|'video'
      status, result_url, poster_url, adopted, created_by, ...)

-- 監修・レビュー履歴（新設。誰が・どの段階で・何を指摘したかの台帳）
reviews(id, target_kind,  -- 'episode'|'asset'
        target_id, stage, -- 'internal'|'medical'|'client'
        verdict,          -- 'ok'|'retake'
        comment, reviewer, created_at)
```

### ステータス機械（話数）

```
脚本入稿 → コンテ中 → コンテOK(社内) → 生成GO → 生成中 → 編集中
        → 監修出し(医学) → 監修OK → クライアント確認(CBC/愛知医大) → 完成
   ※ どの段階でも feedback + ステータス差し戻しでAIワーカーに再投入（M2の再投入をそのまま）
```

天使ちゃんv15.1の教訓を引き継ぐ: **「承認(GO)」と「完成(検収OK)」を絶対に混同させない文言**にする。
本案件はゲートが3つ（社内→医学監修→クライアント）に増えるので、UIに凡例を常設する。

### ステータス機械（設定アセット）

```
案（AI生成たたき） → 社内OK（美術・演出判断） → 監修済（愛知医大の用語・描写チェック）
```

**「監修済」アセットのみ本番カットの参照画像に使える**というガードが本案件の肝
（LDLモンスターの見た目や薬剤キャラの描写が医学的に不適切なまま量産される事故を防ぐ）。

## 5. 制作パイプライン（P0〜P6）

| 工程 | 入力→出力 | 担当 | 天使ちゃん対応 |
|---|---|---|---|
| **P0 脚本取り込み** | 脚本PDF → 整形md＋構造化（シーン/ト書き/セリフ）→ 話数カード自動作成 | AIワーカー | M7の置き換え。ep1〜5は`runs/aichi/scripts/`に完了済 |
| **P1 シリーズ設定** | series_bible → キャラデザ案（三面図・表情集）、美術設定（舞台ボード）、スタイル定義（線・彩度・画角） | AIワーカーが案を量産、人が選定、監修ゲート | M4の拡張。gpt-image-2/nano-banana系で三面図生成 |
| **P2 絵コンテ** | 脚本＋設定 → cuts（カット割り・尺・セリフ・カメラ・コンテ画）＋scenario | AIワーカー生成 → 演出がダッシュボードで修正・承認 | M2/M3そのまま。コンテ画=各カットの静止画1枚（設定を参照画像に） |
| **P3 カット生成** | 承認済みcuts → 静止画（レイアウト）→ 動画（image-to-video） | AIワーカー投入、人はテイク採用 | M4。参照=キャラ三面図＋舞台ボード＋（類似カットがあれば）設定ライブラリ検索(M5) |
| **P4 編集** | 採用テイク → Premiere XML/EDL書き出し → VO（後録り）・BGM・SE・テロップ | 人（アイズ編集） | M6そのまま。VO後録り方式も天使ちゃんと同じ（scenarioにセリフ全文を保持） |
| **P5 監修** | 完パケ候補 → 医学監修（用語・描写）→ CBC/愛知医大確認 | 人。指摘はreviewsに記録→再投入 | 新設。用語対応表(series_bible)がチェックリストの土台 |
| **P6 納品** | 監修OK → 納品パッケージ（本編mp4＋メタ情報） | 人 | deliverables(M6) |

### 一貫性の担保（20話・新規IPで最重要）
1. **定型プロンプト方式（M4）をキャラ全員と主要舞台に適用**: 各アセットに `prompt_en`
   （固定英文）を持たせ、カットプロンプトは「定型部＋シーン差分」の合成で作る。
   尚善で確立したJP/EN対訳・レイヤー合成（`docs/prompt-design.md` / `runs/scene_profiles/`）の流儀を使う。
2. **参照画像は「監修済」アセットに限定**（§4）。
3. **設定ライブラリを育てる（M5の逆順）**: 天使ちゃんは過去動画からライブラリを作ったが、
   本案件は完成カットを都度タグ付けして登録し、後半話数ほど「過去カット参照」で場面一致の
   精度が上がる構造にする。同じ舞台（火山・王城・山道）が20話で繰り返し出るため効果が大きい。
4. **連続性チェック**: 連番話（ep3→4→5は同日の連続シーン）はカット表に「引き継ぎ小物・
   衣装・時間帯」欄を持たせ、AIワーカーがプロンプトへ自動反映する。

### 医学的正確性の担保
- `series_bible.md` の用語対応表を機械チェック化: セリフ・テロップ・プロンプトを
  用語集と突合し、表記揺れ（例: 現1稿の「エゼミチブ」）や未承認の医学表現を検出してから監修に出す。
- 監修指摘は `reviews` に台帳化し、同種指摘の再発をAIワーカーのプロンプト規則に昇格する
  （＝尚善の「レビュー由来の改修」と同じ運用）。

## 6. 数量・スケジュール感（v0.1の見積もり）

- 30〜60秒・会話劇中心 → **1話 8〜15カット、全20話でおよそ200〜300カット**。
  テイク倍率×2〜3で生成ジョブは600〜900本（天使ちゃん実績ベースの検証が8月の宿題）。
- 逆算マイルストーン（納期 2026-11末）:
  - **7月**: 設計確定・アプリのフォーク着工。脚本10本入稿（7月末）。キャラデザ・美術設定の案出し開始
  - **8月**: 設定の監修OK取得 → **パイロット1話をP0〜P6通しで完成**させ、監修フローと
    テイク倍率・所要クレジットを実測
  - **9〜10月**: 量産（週2〜3話ペース。カット生成はAIワーカー並行数で調整）
  - **11月**: 検収・リテイク・納品バッファ
- リスク: 脚本後半10本の入稿時期／監修の往復回数／クレジット・アカウント体制
  （天使ちゃんβでも未決のまま。§8）。

## 8. 未決事項（ユーザー/チーム判断待ち）

1. **アプリの形**: 新規フォーク（本設計の推奨）か、天使ちゃんスタジオのマルチ作品化か。
2. **納品仕様**: 画角（16:9 か 9:16 か両方か）・解像度・尺の厳密値・テロップ/ナレーションの有無。
   CBC経由のため放映先仕様の確認が必要（生成モデル選定とコンテの画面設計に直結）。
3. **生成モデルの本命**: 天使ちゃんはSeedance 2.0 (9:16/720p)。本案件のセルルック・
   ファンタジー画風で何を使うかはP1のスタイル検証で決める（fnfはkling-3/veo/wan等も投入可能）。
4. **クレジット・アカウント体制**: 天使ちゃんβと同じ課題（現状は個人アカウント紐づけ）。
5. **監修の窓口と往復ルール**: 愛知医大側の監修者・回数・所要日数。CBCの確認が別枠か。
6. **VO体制**: 声優後録り前提でよいか（scenario にセリフ全文を保持する設計はそれを前提）。
7. **キャラデザのジャッジ役**: G.O.A.T.案件でも未決のまま指摘されている論点。本案件は
   愛知医大の監修があるぶん、社内の一次ジャッジを誰が持つか先に決めたい。
